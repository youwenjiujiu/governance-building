# 酒店行业场景定义

> **Container**: `hotel`
> **定位**: 楼宇远程运营平台酒店行业 Container，覆盖酒店全业态机电系统的远程运维管理。
> **适用场景**: 经济型/商务型/五星级/度假型酒店的 HVAC、GRMS 客房控制、热水系统、泳池/SPA、洗衣房、厨房、宴会厅等全系统远程运营。

| 洗碗机温度不足 | KIT_Dishwash_Temp < 65°C | 警告 | 消毒不达标，检查加热器/蒸汽供应 |
| HACCP 冷链断链 | KIT_HACCP_Temp_N 超出安全范围持续 >15min | 严重 | 记录偏差报告，评估食材是否可用 |

### 2.5 HVAC 告警（酒店特有补充）

> 冷站、水泵、AHU/MAU、FCU 的通用告警规则参考 `containers/office-hvac/scenario.md` 2.1~2.6 节。以下为酒店业态特有的 HVAC 告警。

| 告警名称 | 条件 | 严重度 | 处理建议 |
|---------|------|--------|---------|
| 大堂温度偏差 | 大堂 Zone_Temp > 26°C 或 < 20°C 持续 10min | 警告 | 大堂是酒店门面，优先处理；检查 AHU/热风幕 |
| 大堂热风幕未联动 | 室外温度 <5°C 且 大门频繁开启 且 Lobby_AirCurtain=OFF | 警告 | 开启热风幕，防止冷风渗透 |
| 宴会厅温度偏差 | BQT_Zone_Temp 偏离设定 >3°C 持续 15min 且 有活动进行 | 严重 | 检查 AHU 运行/冷热水供应，宴会期间客诉风险极高 |
| 宴会厅 CO2 偏高 | BQT_CO2 > 1200ppm 持续 10min | 警告 | 增加新风量，必要时开启门窗辅助通风 |
| 泳池区域结露 | Pool_DewPoint 接近壁面温度（差 <2°C） | 警告 | 增加除湿 AHU 运行功率，提高壁面温度 |
| 客房走廊温度普遍异常 | 同层 ≥5 间客房 Room_Temp 偏差 >3°C | 严重 | 非个别 FCU 问题，检查该层管路/水力平衡/AHU |

### 2.6 组合告警（关联分析）

| 告警名称 | 条件 | 严重度 | 判断逻辑 |
|---------|------|--------|---------|
| 全楼热水危机 | 所有水箱 Tank_Temp_Mid < 45°C + 所有热源故障/停机 | 紧急 | 客人无热水，启动应急热源（如电加热）；若无法恢复需通知客人 |

| 夜间能耗异常 | 凌晨 2:00~5:00 总电耗 > 白天均值 50% + 入住率 <50% | 警告 | 可能有设备未按排程关闭，或设备故障持续运行 |
| VIP 客房环境不达标 | VIP 房间 Room_Temp 偏离偏好 >2°C 或 Room_RH >70% | 严重 | VIP 客人入住前/中需优先保障，通知客房部+工程部 |

## 4. 运营场景剧本（Playbook）

### 场景1：VIP客人入住全流程联动

**触发条件**: PMS 推送 VIP 预抵信号（入住前 2~4 小时），包含客人偏好数据

**涉及设备**: 目标客房 RCU（空调/灯光/窗帘）+ 走廊照明 + 电梯 + 大堂灯光

**操作步骤**:

```
T-4h  PMS 推送 VIP 预订信息：
      ├── 读取客人偏好：温度偏好（如 22°C）、灯光偏好（暖光）、枕头类型等
      ├── 客房状态确认：房间已清洁（INSPECTED）→ 触发 VIP 预设流程
      └── 通知客房部准备 VIP 欢迎布置

T-60min  客房环境预设：
         ├── 空调启动至 VIP 偏好温度（如 22°C，制冷或制热模式自动判断）
         ├── 风速设定 LOW（安静模式）
         ├── 灯光设定为 VIP 欢迎场景（暖光 60%亮度，夜灯开启）
         ├── 窗帘设定：纱帘半开，遮光帘全开
         ├── 取电开关远程强制通电
         └── 迷你吧确认正常运行（温度 2~8°C）

T-30min  环境检查：
         ├── 确认房间温度已达设定值（±1°C）
         ├── 确认空气质量（无异味，新风已充分换气）
         └── 如温度未达标 → 告警通知工程部优先处理

T+0  客人到达（前台 Check-in）：
     ├── PMS 状态变更为 OCCUPIED
     ├── 大堂播放欢迎提示（如有LED系统联动）
     ├── 电梯智能调度（可选：VIP 专梯优先）
     └── 走廊灯光自动增亮（客人所在楼层）

T+check-in  客人进入房间：
            ├── 门磁检测到开门 → 确认客人到达
            ├── 灯光自动切至欢迎场景
            ├── 电视自动开机显示欢迎画面（如有联动）
            └── 系统切至"在住"模式，客人可自主调节

住宿期间：
├── 系统学习客人使用习惯（如常设温度、灯光偏好）
├── 退出房间（卡拔出）→ 15min 后进入节能模式（温度回调 ±3°C）
├── 回到房间（卡插入）→ 恢复至客人偏好温度
└── 夜间 23:00 自动切至安静模式（风速 LOW，灯光渐暗）
```

**判断逻辑**:
- VIP 等级决定预冷/预热提前时间（V1: 30min, V2: 60min, V3: 120min）
- 室外温度 >35°C 或 <0°C 时，预冷/预热提前量额外增加 30min
- 如客房 FCU 故障 → 自动升级告警 + 提前通知前台换房

**预期结果**:
- 客人进入房间时温度已达偏好设定（±1°C）
- 灯光场景自动呈现，无需客人手动操作
- VIP 投诉率 < 0.5%

### 场景2：大型婚宴全流程（准备→进行→散场）

**触发条件**: 宴会预约系统确认婚宴（300~500人，18:00~21:00）

**涉及设备**: 宴会厅 AHU + 灯光控制器 + 冷站（可能加机）+ 厨房排风 + 走廊/电梯空调

**操作步骤**:

```
T-5h（13:00）  准备阶段：
              ├── 宴会厅隔断墙状态确认（全开/分厅）
              ├── 灯光系统通电，切至布场模式（全亮白光）
              ├── AHU 不启动（布场人员少，开门通风即可）
              └── 厨房备餐启动，排风系统开启

T-2h（16:00）  预冷/预热启动：
              ├── 宴会厅 AHU 启动（送风温度设定：夏季 14°C/冬季 38°C）
              ├── 冷站评估当前负荷 → 判断是否需加机
              │   ├── 入住率 >80% + 婚宴 500人 → 通常需加 1~2 台冷机
              │   └── 提前预警冷站团队准备
              ├── 新风阀开至 50%（预通风排异味）
              ├── 走廊/电梯厅空调增强（宾客通行区域）
              └── 温度目标：宴会厅 24~25°C，走廊 25°C

T-30min（17:30）  迎宾准备：
                  ├── 灯光切至迎宾场景（暖光，重点照明入口/签到区）
                  ├── AHU 全速运行，新风 100%
                  ├── 确认宴会厅温度达标（24°C ±1°C）
                  ├── 背景音乐系统联动（如有）
                  └── 大堂增强照明（婚宴引导灯光）

T+0（18:00）  婚宴开始：
             ├── 灯光切至宴会场景（暖光 70%，舞台区重点照明）
             ├── AHU 正常 PID 控制，送风温度根据回风温度调节
             ├── CO2 监控启动（500 人场景 CO2 上升快）
             │   ├── CO2 < 800ppm：维持当前新风量
             │   ├── CO2 800~1200ppm：新风量增至 100%
             │   └── CO2 > 1200ppm：告警，开启辅助通风
             ├── 厨房全力排风（婚宴上菜高峰）
             └── 实时监控宴会厅温度/湿度

T+1.5h（19:30）  宴会高峰：
                 ├── 菜品陆续上桌，厅内热负荷增大（人体+菜品+灯光）
                 ├── 送风温度可能需下调 1~2°C
                 ├── 关注冷站负荷（婚宴+客房叠加高峰）
                 └── 如温度偏高 → 加大送风量/降低送风温度

T+end（21:00）  婚宴结束：
               ├── 灯光切至散场模式（全亮，引导宾客离场）
               ├── AHU 维持运行（大量人员离场，维持通风）
               ├── 走廊/电梯厅空调维持增强模式
```

### 场景5：热水系统军团菌防控冲洗

**触发条件**: 定期（每周一次）或 触发条件（水箱下部温度 <50°C 持续 >24h）

**涉及设备**: 热水锅炉/热泵 + 储热水箱 + 循环泵 + 各层回水管路

**操作步骤**:

```
1. 冲洗准备（通常安排在凌晨 02:00~05:00，用水低谷期）
   ├── 通知客房部（期间热水温度可能短暂偏高）
   ├── 确认恒温混水阀功能正常（防止烫伤）
   ├── 锅炉/热泵满负荷启动
   └── 目标：将水箱温度提升至 ≥65°C

2. 高温冲洗执行
   ├── 储热水箱加热至 65~70°C（顶部先达标，逐步全箱达标）
   ├── 循环泵全速运行
   ├── 各层回水温度监控：
   │   ├── 目标：所有层回水温度 ≥60°C
   │   ├── 持续时间：每个末端点 ≥60°C 维持 ≥30min
   │   └── 如某层回水温度迟迟上不来 → 检查阀门/管路堵塞
   ├── 总管供水温度维持 65~70°C
   └── 冲洗持续 1~2h（视管网规模）

3. 恢复正常
   ├── 锅炉/热泵恢复正常出水设定（55~60°C）
   ├── 水箱温度自然回落至正常存储温度
   ├── 恒温混水阀确认出水温度恢复至 42~50°C
   ├── 循环泵恢复正常频率
   └── 各层末端出水温度抽检

4. 记录
   ├── 生成军团菌防控冲洗报告
   ├── 记录各层回水达标时间
   ├── 记录水箱最高温度/持续时间
   └── 归档至合规记录（部分地区法规要求）
```

**判断逻辑**:
- 水箱底部温度长期 <50°C → 军团菌繁殖风险区间（25~45°C）
- 回水最远端温度是判断冲洗是否充分的关键指标
- 夏季热泵 COP 高，优先使用热泵加热；冬季可能需要锅炉辅助
- 冲洗期间恒温混水阀是防烫伤的最后防线，必须确认可靠

**预期结果**:
- 全系统水温 ≥60°C 维持 ≥30min
- 所有层回水温度达标记录完整
- 无客人烫伤事故
- 合规记录归档

### 场景6：泳池水质异常处理

**触发条件**: 水质在线监测仪检测到 pH/余氯/浊度超出安全范围

**涉及设备**: 泳池水质监测仪 + 投药泵 + 循环泵 + 砂缸过滤器

**操作步骤**:

```
情况A：余氯偏低（Pool_Cl < 0.3 mg/L）
├── 1. 系统自动增加次氯酸钠投药速率（+30%）
├── 2. 15min 后复检 → 如仍偏低 → 继续增加（+50%）
├── 3. 30min 后仍偏低 → 告警通知泳池管理员
├── 4. 人工检查：药液余量/投药泵工作/探头准确性
├── 5. 如 60min 无法恢复 → 关闭泳池，人工投药
└── 6. 余氯恢复至 0.3~1.0 mg/L → 恢复开放

情况B：pH 偏移（pH < 7.0 或 > 7.8）
├── 1. 系统自动调节 pH 投药泵（盐酸降 pH/碳酸钠升 pH）
├── 2. 投药后 15min 复检
├── 3. 如 pH < 6.5 或 > 8.5 → 紧急关闭泳池
├── 4. 人工检查投药系统/管路/探头校准
├── 5. pH 恢复至 7.2~7.6 → 稳定 30min → 恢复开放
└── 6. 生成偏差报告

情况C：浊度超标（Pool_Turb > 2.0 NTU）
├── 1. 循环泵切至全速
├── 2. 检查砂缸压差 → 如 >70kPa → 执行反冲洗
├── 3. 反冲洗流程：停循环泵 → 切换阀门至反冲洗 → 全速冲洗 5min → 恢复正洗
├── 4. 反冲洗后补水至正常液位
├── 5. 如浊度 >5.0 NTU → 关闭泳池 → 超氯处理+絮凝剂+强制循环
├── 6. 浊度降至 <1.0 NTU → 恢复开放
└── 7. 生成水质异常报告

情况D：多指标同时异常
├── 1. 立即关闭泳池
├── 2. 全面水质检测（人工取样化验）
├── 3. 可能原因排查：过多泳客/探头故障/投药系统故障/水源污染
├── 4. 根据化验结果制定处理方案
└── 5. 所有指标恢复正常 → 稳定运行 1h → 恢复开放
```

**判断逻辑**:
- 单指标轻微偏移 → 自动调节，不关闭泳池
- 单指标严重偏移 → 关闭泳池，自动+人工处理
- 多指标同时异常 → 立即关闭，全面排查
- 探头校准记录超过 7 天 → 建议人工校准后再信任读数

**预期结果**:
- 轻微偏移 30min 内自动恢复
- 严重偏移 2h 内恢复
- 零因水质导致的客人健康事故
- 完整水质异常处理记录

### 场景7：厨房燃气泄漏应急

**触发条件**: KIT_Gas_Alarm = ALARM 或 KIT_Gas_Conc > 25%LEL

**涉及设备**: 燃气切断阀 + 厨房排风 + 厨房补风 + 消防系统 + HVAC

**操作步骤**:

```
T+0s   燃气泄漏检测：
       ├── 系统接收燃气报警信号
       ├── 生成紧急告警（推送至：工程主管/总值班/安保/厨师长）
       ├── 声光报警器启动（厨房区域）
       └── 记录报警时间、浓度值、报警位置

T+3s   自动应急响应（全自动，不等人工确认）：
       ├── **立即关闭燃气紧急切断阀**（KIT_Gas_Valve → CLOSED）
       ├── **启动厨房排风机全速运行**（KIT_ExhFan_Hz → 50Hz）
       ├── **启动补风机全速运行**（强制通风稀释）
       ├── **切断厨房非防爆电器电源**（防止电火花引燃）
       └── 厨房区域 HVAC 系统：新风阀全开、回风阀全关（防止燃气扩散至其他区域）

T+1min 人员疏散：
       ├── 厨房人员立即撤离至安全区域
       ├── 安保人员到场警戒（禁止无关人员进入）
       ├── 通知消防控制室
       └── 如浓度 >50%LEL → 拨打 119

T+5min 浓度监测：
       ├── 持续监控 KIT_Gas_Conc
       ├── 如浓度下降（排风稀释有效）→ 继续排风
       ├── 如浓度不降反升 → 可能仍在泄漏，确认切断阀是否到位
       └── 通知燃气公司到场检修

T+15~30min 浓度降至安全范围（<5%LEL）：
           ├── 工程人员穿戴防护设备进入检查
           ├── 检漏仪排查泄漏点（管接头/阀门/灶具接口）
           ├── 修复泄漏点
           └── 管路打压测试

T+修复后 恢复运行：
         ├── 确认浓度持续 <2%LEL 超过 15min
         ├── 燃气切断阀恢复开启（需手动确认或远程+双人授权）
         ├── 逐一恢复灶具点火
         ├── 排风系统维持全速 30min
         ├── 厨房恢复正常运行
         └── 生成事故报告
```

**判断逻辑**:
- 浓度 10~25%LEL → 预警级别，加强通风+巡查，不切断燃气
- 浓度 >25%LEL → 报警级别，自动切断燃气
- 浓度 >50%LEL → 极度危险，119 + 全面疏散
- 燃气切断阀恢复需双人授权（防止误恢复）

**预期结果**:
- 燃气切断响应时间 <5s
- 零人员伤亡
- 厨房停摆时间 <2h（轻微泄漏）/ <4h（较严重泄漏）
- 完整事故报告归档

### 场景8：酒店停电应急

**触发条件**: 市电中断 / 变压器故障 / 主开关跳闸

**涉及设备**: 应急发电机 + UPS + 全酒店电气系统 + BMS + GRMS + 消防 + 安防

**操作步骤**:

```
T+0s   停电检测：
       ├── BMS 检测到市电中断信号
       ├── UPS 自动切换至电池供电（BMS/安防/消防/应急照明）
       ├── 全酒店非应急回路断电
       └── 紧急告警推送（总工/总值班/前台/安保/工程部全员）

T+10~15s  发电机自动启动：
          ├── 应急柴油发电机自动启动+暖机
          ├── ATS（自动转换开关）检测发电机电压/频率稳定
          └── ATS 自动切换至发电机供电

T+30~60s  应急供电恢复（按优先级分批送电）：
          ├── 第1批（立即）：消防系统/应急照明/安防系统/电梯（迫降至1层）
          ├── 第2批（+30s）：BMS/GRMS 服务器/通讯系统/前台系统
          ├── 第3批（+60s）：客房走廊照明/大堂基本照明
          ├── 第4批（+2min）：热水循环泵/冷库制冷（食材保护）
          └── 非优先负载（不送电）：宴会厅/商场/泳池/洗衣房/部分空调

T+5min 状况评估：
       ├── 确认发电机运行稳定（电压/频率/负载率）
       ├── 评估停电原因（市电故障/内部故障）
       ├── 联系供电局确认预计恢复时间
       ├── 如预计恢复 >1h → 通知客人
       └── 发电机油量检查（是否够用）

停电期间管理：
├── GRMS 降级运行：客房仅保留基本照明+取电开关+门锁+SOS
├── HVAC：仅保留关键区域最低限度通风
│   ├── 客房空调全停（发电机容量不足以带全部空调）
│   ├── 大堂保留 1 台 AHU 低速运行（如发电机容量允许）
│   └── 冷库/机房通风优先
├── 电梯：保留 1~2 台客梯运行
├── 热水：循环泵运行，锅炉/热泵视发电机容量决定
├── 前台/PMS：UPS + 发电机保障
└── 每 30min 更新一次客人通知

T+恢复 市电恢复：
        ├── ATS 检测市电稳定 →自动/手动切换回市电
        ├── 分批恢复非优先负载
        ├── 冷站重新启动（注意启动电流，错峰启动各冷机）
        ├── GRMS/HVAC 全面恢复
        ├── 电梯全部恢复
        ├── 发电机卸载 → 冷却运行 5min → 停机
        └── 发电机补油+检查记录
```

**判断逻辑**:
- 停电 <30min → 不通知客人，静默恢复
- 停电 30min~2h → 前台/APP 通知客人，提供基本服务保障
- 停电 >2h → 全面应急预案，考虑为客人提供替代方案
- 发电机负载 >85% → 进一步削减非必要负载
- 发电机油量 <4h → 安排紧急加油

**预期结果**:
- 发电机 15s 内启动并供电
- 消防/安防/应急照明零中断（UPS 保障）
- 客人基本生活功能保障（照明/热水/电梯/门锁）
- 食材安全（冷库温度上升 <3°C/h）

### 场景9：客房投诉处理（太冷/太热）

**触发条件**: 客人致电前台投诉房间温度不适 / APP 投诉 / 客房温度告警

**涉及设备**: 目标客房 RCU + FCU + 冷冻水/热水系统

**操作步骤**:

```
1. 接收投诉
   ├── 前台/APP 记录：房号、客人姓名、投诉内容（太冷/太热/无风）
   ├── 系统自动拉取该客房实时数据：
   │   ├── 当前房间温度 vs 设定温度
   │   ├── 空调模式/风速/水阀状态
   │   ├── 取电开关状态/人体感应状态
   │   ├── 窗户状态（是否开窗）
   │   └── 该楼层其他客房温度（判断是否普遍问题）
   └── 初步判断：个别问题 vs 系统性问题

2. 远程快速处理（<2min）
   ├── 情况A：设定温度不合理
   │   ├── 如客人设定 16°C（可能误操作）→ 电话引导调至 22~24°C
   │   └── 远程辅助调节温度设定
   │
   ├── 情况B：窗户开启
   │   ├── Window_Status=OPEN → 引导客人关窗
   │   └── 系统可能已自动关闭空调（窗户联锁）
   │
   ├── 情况C：空调模式错误
   │   ├── 夏季制热/冬季制冷 → 远程切换正确模式
   │   └── 风速设定 LOW → 建议切至 HIGH 快速调温
   │
   ├── 情况D：水阀/FCU 问题（远程可见）
   │   ├── 水阀开度 100% 但温度不变 → 可能水阀卡死或冷热水供应问题
   │   ├── 尝试远程重启 FCU（关→开）
   │   └── 如无效 → 派工程师现场处理
   │
   └── 情况E：系统性问题
       ├── 同楼层多房间同一问题 → 检查楼层管路/阀门/水力平衡
       ├── 全楼普遍问题 → 检查冷站/热源运行状态
       └── 提升至工程主管处理

3. 现场处理（如远程无法解决）
   ├── 工程师 15min 内到房间
   ├── 检查 FCU 风机/水阀/滤网
   ├── 检查管路温度（确认冷热水到位）
   ├── 如 FCU 故障 → 临时提供移动空调/电暖器
   └── 如需修复时间 >30min → 建议前台换房

4. 跟进
   ├── 30min 后回访客人确认温度是否改善
   ├── 记录投诉处理过程和结果
   ├── 生成工单（如需设备维修）
   └── 纳入月度投诉分析报告
```

**判断逻辑**:
- 单房投诉 + 其他房间正常 → 个别 FCU 问题或客人操作问题
- 多房投诉 + 同侧/同层 → 楼层管路/水力平衡问题
- 多房投诉 + 分布广泛 → 冷站/热源系统问题
- VIP 客人投诉 → 优先级最高，工程主管亲自处理

**预期结果**:
- 远程可解决的问题 5min 内响应
- 现场处理 15min 内工程师到达
- 客人满意率 >95%
- 同一房间重复投诉率 <5%

### 场景10：淡季/旺季模式切换

**触发条件**: 入住率持续变化达到阈值 / 运营部手动切换 / 节假日日历自动触发

**涉及设备**: 全酒店 HVAC + GRMS + 热水 + 泳池 + 洗衣房 + 厨房 + 公区

**操作步骤**:

```
旺季模式（入住率 >75% 或节假日/展会/旅游旺季）：
├── HVAC
│   ├── 冷站/热源保持最大可用台数（备用机随时就绪）
│   ├── 公区 AHU 全速运行
│   ├── 走廊/电梯厅空调增强
│   └── 宴会厅/会议厅按预约动态调度
│
├── GRMS
│   ├── 预抵预冷/预热提前量增加（全部房间 30min 提前）
│   ├── 空房节能模式阈值收紧（更快进入节能）
│   └── 夜间走廊照明维持较高水平
│
├── 热水
│   ├── 水箱全部满储热
│   ├── 热源设备全部待命
│   ├── 循环泵全速运行（减少客人等待热水时间）
│   └── 军团菌冲洗频率维持每周一次
│
├── 泳池
│   ├── 全时段开放
│   ├── 水质监测频率加密（每 5min 一次）
│   ├── 救生员/服务增配
│   └── 除湿 AHU 全速
│
├── 厨房/餐厅
│   ├── 全部餐厅开放
│   ├── 排风系统全时段运行
│   ├── 冷库加强监控（食材周转加快）
│   └── 早餐时段延长
│
└── 能耗管控
    ├── 放宽节能策略（优先保障服务品质）
    ├── 加强能耗对标监控
    └── 确保不超电力合同容量

淡季模式（入住率 <40% 持续 >3 天）：
├── HVAC
│   ├── 冷站/热源按最低需求运行（减机/减泵）
│   ├── 关闭部分楼层空调（整层空房时）
│   ├── 公区 AHU 降频运行
│   └── 宴会厅/会议厅默认全关
│
├── GRMS
│   ├── 集中安排客人至相邻楼层/相邻房间（PMS 协调）
│   ├── 非入住楼层 GRMS 切至深度节能（温度回落至 28°C/14°C）
│   ├── 非入住楼层走廊照明降至 30%
│   └── 减少走廊空调运行时间
│
├── 热水
│   ├── 减少水箱储热量（仅维持 50~60% 储热）
│   ├── 部分热源设备停用
│   ├── 循环泵降频运行
│   └── 军团菌冲洗维持（管路不可停止循环）
│
├── 泳池
│   ├── 缩短开放时段（如仅 08:00~20:00）
│   ├── 降低水温设定 1~2°C
│   ├── 循环泵降频
│   └── 如入住率 <15% → 考虑暂时关闭泳池
│
├── 厨房/餐厅
│   ├── 关闭部分餐厅（仅保留全日餐厅）
│   ├── 关闭餐厅的排风/空调关闭
│   ├── 厨房缩减运营区域
│   └── 冷库可关闭备用库
│
└── 能耗管控
    ├── 启用激进节能策略
    ├── 逐日对比入住率 vs 能耗
    ├── 目标：单间夜能耗降低 20~30%
    └── 月度汇总节能效果报告
```

**判断逻辑**:
- 入住率 >75% → 旺季模式
- 入住率 40~75% → 标准模式
- 入住率 <40% → 淡季模式
- 切换需考虑客人体验 vs 节能平衡
- 淡季集中排房可大幅降低整层空调/照明能耗
- 节假日即使入住率未达旺季标准也切至旺季模式（服务品质优先）

**预期结果**:
- 旺季：零因设备能力不足导致的投诉
- 淡季：整体能耗降低 25~40%
- 模式切换对客人无感知影响

## 6. 与PMS集成

### 6.1 PMS 对接概述

酒店管理系统（PMS, Property Management System）是酒店运营的核心系统，掌握客房状态、客人信息、预订数据等。BMS/GRMS 与 PMS 的集成是酒店智能化的关键。

```
PMS ←→ 中间件/API 网关 ←→ BMS/GRMS/能源管理

数据流向：
├── PMS → BMS/GRMS（下行）：
│   ├── 客房状态变更（预抵/入住/退房/维修/锁房）
│   ├── VIP 等级和偏好参数
│   ├── 宴会/会议预约信息
│   ├── 餐厅营业时段调整
│   └── 入住率预测（次日/本周）
│
├── BMS/GRMS → PMS（上行）：
│   ├── 客房实时环境数据（温度/湿度）
│   ├── 客房设备状态（DND/MUR/门磁/SOS）
│   ├── 客房能耗数据（分房计量）
│   ├── 设备故障影响客房信息
│   └── 迷你吧开启记录
│
└── 双向同步：
    ├── 房间状态（BMS 维修状态 ↔ PMS 房态）
    └── 告警/工单（BMS 设备故障 → PMS 服务工单）
```

### 6.2 客房状态同步

| PMS 房态 | BMS/GRMS 映射动作 | 触发时机 |
|---------|-------------------|---------|
| `RESERVED`（已预订） | 无操作，房间维持空房节能模式 | 客人预订时 |
| `PRE_ARRIVAL`（预抵） | 预冷/预热启动 + 灯光欢迎场景 + 通电 | 预计入住前 30~60min |
| `CHECKED_IN`（已入住） | 切至在住模式，客人可自主控制 | 前台办理入住 |
| `OCCUPIED`（在住） | 正常服务模式 | 入住期间 |
| `CHECKED_OUT`（已退房） | 空调关闭/节能 + 灯光全关 + 窗帘全开 + 断电 | 前台办理退房 |
| `DIRTY`（待清洁） | 通风模式（低速送风换气） | 退房后 |
| `CLEANING`（清洁中） | 全亮灯光 + 低速通风 + 通电 | 客房部进入清洁 |
| `INSPECTED`（已检查） | 空房节能模式 | 清洁完成 |
| `OUT_OF_ORDER`（维修中） | 按工程需求控制，不参与排房 | 工程部标记 |
| `OUT_OF_SERVICE`（停用） | 全部关闭，仅保留防冻 | 淡季关闭楼层 |

### 6.3 入住/退房联动

#### 入住联动流程

```
PMS 事件：CHECKED_IN(房号=1208, 客人=张三, VIP=V2)
    │
    ├── API 推送至 GRMS 中间件
    │
    ├── GRMS 执行联动：
    │   ├── Room 1208 RCU：
    │   │   ├── AC_Mode → AUTO
    │   │   ├── Room_Temp_SP → 23°C（V2 偏好）
    │   │   ├── Fan_Speed → LOW
    │   │   ├── Light_Scene → WELCOME
    │   │   ├── Curtain_Pos → 50%
    │   │   ├── KeyCard_Override → FORCE_ON
    │   │   └── Room_State → OCCUPIED
    │   │
    │   └── 走廊灯光：12层走廊照明确保 100%
    │
    ├── BMS 联动（如需）：
    │   ├── 确认 12 层冷冻水/热水供应正常
    │   └── 走廊 FCU 确认正常运行
    │
    └── 事件记录：
        ├── AuditLog 记录入住联动
        └── 入住联动耗时统计
```

#### 退房联动流程

```
PMS 事件：CHECKED_OUT(房号=1208)
    │
    ├── GRMS 执行联动：
    │   ├── Room 1208 RCU：
    │   │   ├── AC_OnOff → OFF（或切至节能模式）
    │   │   ├── Room_Temp_SP → 28°C（夏季节能）/ 16°C（冬季防冻）
    │   │   ├── Light_Master → OFF
    │   │   ├── Curtain → 全开（通风+日光消毒）
    │   │   ├── KeyCard_Override → AUTO（断电）
    │   │   └── Room_State → CHECKOUT
    │   │
    │   └── 客房能耗快照：
    │       ├── 记录本次入住能耗（入住~退房 Room_kWh 差值）
    │       └── 纳入间夜能耗统计
    │
    └── 等待清洁 → PMS 推送 CLEANING → 通电+全亮
```

## 7. 与 base 层的对接说明

### 7.1 使用的 base 层实体

### 7.4 base schema 不够用的地方（需扩展的部分）

以下标注酒店行业在当前 base schema 中放不下、需要扩展或新增的能力：

| 领域 | 缺口描述 | 当前 base 状态 | 建议扩展方式 |
|------|---------|---------------|-------------|
| **事件驱动排程** | 酒店大量排程不是固定时间表，而是由 PMS 事件（入住/退房/宴会预约）动态触发 | base.Schedule 仅支持 `WEEKLY`/`YEARLY` 固定排程 | 新增 `EVENT_DRIVEN` 排程类型，增加 `trigger_event_type` / `pre_condition_minutes` 字段 |
| **外部系统集成** | 酒店强依赖 PMS，BMS 需要与 PMS 实时双向通信 | base 无外部系统集成实体 | 新增 `ExternalIntegration` 实体，支持配置 API 端点/认证/字段映射/同步状态 |
| **客房状态机** | 客房有复杂的状态流转（空房→预抵→入住→外出→在住→退房→清洁→待查→空房） | base.Zone 仅有通用 `status (ACTIVE/INACTIVE)` | 新增 `pms_room_status` 字段或 `GuestRoomState` 实体，支持客房全生命周期状态 |
| **水质监测** | 泳池水质指标（pH/余氯/ORP/浊度）是连续传感器数据但也需要人工化验补充 | base.Point 支持传感器但无"人工+自动混合采集"机制 | Point 增加 `manual_entry_enabled` 字段，或新增 `ManualReading` 实体关联 Point |
| **食品安全 HACCP** | 厨房冷链/出品温度有严格的 HACCP 合规要求（偏差记录/纠偏行动/时间窗口） | base.ComplianceRule 可配置规则，但无 HACCP 专用偏差记录流程 | 新增 `HaccpDeviationLog` 实体，包含偏差描述/纠偏行动/签字确认 |
| **VIP 偏好模型** | 酒店需要存储和下发客人偏好（温度/灯光/窗帘/迷你吧），并持续学习更新 | base 无客人/住客概念 | 新增 `GuestPreference` 实体，或在 User 实体的 metadata 中扩展（但客人不是系统 User） |
| **SpaceType 枚举不够** | 酒店需要更多空间类型（SPA/泳池/宴会厅/行政酒廊/健身房/洗衣房） | base.SpaceType 已有 GUEST_ROOM/KITCHEN/DINING 但缺少 SPA/POOL/BANQUET_HALL/LAUNDRY/FITNESS/CLUB_LOUNGE | 扩展 SpaceType 枚举新增上述值 |
| **SystemType 枚举不够** | 酒店有 GRMS/DHW/POOL 等独立子系统 | base.SystemType 缺少 GRMS/DHW/POOL/LAUNDRY/KITCHEN_EXHAUST | 扩展 SystemType 枚举 |
| **EquipmentType 枚举不够** | 酒店有 RCU/热泵热水/投药泵/砂缸过滤器/油烟净化器等特有设备 | base.EquipmentType 缺少上述类型 | 扩展 EquipmentType 枚举新增 RCU/ASHP_DHW/DOSING_PUMP/SAND_FILTER/FUME_PURIFIER 等 |
| **军团菌合规** | 热水系统军团菌防控是酒店强制合规要求，需要周期冲洗记录+温度达标证据链 | base.ComplianceRule 可配置但无冲洗记录模板 | 新增 `LegionellaFlushLog` 实体，记录各层回水达标时间/水箱温度曲线 |
| **分房能耗计量** | 酒店需要按客房分别计量能耗（间夜 kWh），与 PMS 房态关联统计 | base.EnergyMeter 支持分户计量但无"间夜"概念 | 新增 `RoomNightEnergy` 实体，关联 check-in/check-out 时间段的能耗读数差值 |
| **多场景灯光** | 酒店灯光场景复杂（大堂日/夜、餐厅早/午/晚、宴会厅多模式） | base.Point 可控制灯光开关/调光，但无"场景"概念 | 新增 `LightingScene` 实体或在 Scenario 中细化灯光动作参数 |
| **宴会动态联动** | 宴会事件需要与 HVAC/灯光/隔断墙/音响等多系统联动，且按时间线分阶段 | base.Scenario 支持触发+动作，但缺少"阶段性时间线"概念 | Scenario 的 actions 增加 `timeline` 支持分阶段动作（如 T-2h/T-30min/T+0/T+end） |
| **门磁/取电/DND 等弱电点位** | GRMS 的点位语义（门磁/取电卡/DND/MUR/SOS）在通用 Brick 分类中没有对应 | base.Point 的 `brick_class` 和 `haystack_tags` 无 GRMS 专用语义 | 需要在 Point 的 tags 体系中新增 GRMS 专用标签（如 `door-contact`/`keycard-switch`/`dnd-indicator`/`sos-alarm`） |
