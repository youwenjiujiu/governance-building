# Building Operations 补充维度

> 前两份文档按「楼宇类型」和「子系统」两个维度穷举了场景。
> 本文补充所有**横切维度** — 这些维度与楼宇类型正交，任何一栋楼都会被这些维度影响。

---

## 维度一：地理与气候（Geography & Climate）

同一种楼宇，在不同气候区的运营逻辑完全不同。

### 气候分区

| 气候类型 | 代表城市 | HVAC 特征 | 特殊挑战 |
|---------|---------|-----------|---------|
| **湿热带** | 新加坡、曼谷、广州、迈阿密 | 制冷为主，365天运行，除湿是核心 | 冷凝水量大、发霉、腐蚀加速 |
| **干热带** | 迪拜、利雅得、拉斯维加斯 | 制冷为主，蒸发冷却有效 | 沙尘暴（滤网频繁更换）、太阳辐射极强 |
| **温带大陆（冷冬热夏）** | 北京、首尔、芝加哥 | 冬季供暖+夏季制冷，过渡季自然通风 | 供暖/制冷切换、防冻、冬季结露 |
| **温带海洋** | 伦敦、悉尼、旧金山 | 冷热需求均不极端 | 潮湿/霉变、海风腐蚀（沿海） |
| **寒带/亚寒带** | 莫斯科、哈尔滨、赫尔辛基 | 供暖为主（6-8个月）| 防冻是生命线、管道保温、融雪 |
| **高海拔** | 拉萨、昆明、波哥大 | 制冷需求小，紫外线强 | 空气稀薄→设备降容、紫外线老化 |
| **热带季风** | 孟买、雅加达、马尼拉 | 全年制冷 + 雨季排水 | 暴雨/台风防汛、雨季霉变 |

### 气候对运营的具体影响

#### 供暖系统（寒冷地区独有）

```
├── 集中供热（市政热力管网/区域锅炉房）
├── 采暖散热器/地暖
├── 防冻保护（管道/水箱/消防管道伴热带）
├── 融雪系统（屋顶/坡道/人行道）
├── 冬季新风预热（热回收转轮/板式换热器）
├── 冬季加湿（干冷气候湿度可降至 10%）
├── 供暖计量（热量表，按户/按面积/按流量）
└── 锅炉房管理（燃气/燃油/电，排放监测）
```

**产品启示**：供暖系统在热带毫无意义，但在寒带是最大的能耗和运营关注点。产品必须按气候区适配。

#### 防灾类型（因地域而异）

| 灾害类型 | 高风险区域 | 对楼宇运营的影响 |
|---------|-----------|----------------|
| **台风/飓风** | 东南沿海、加勒比、日本 | 加固外围护、排水加强、提前关闭户外设备、应急发电 |
| **地震** | 日本、台湾、加州、土耳其 | 震后自动检测（电梯/燃气/结构）、避难层启用、自动关闭燃气 |
| **洪水/内涝** | 低洼城市、河流沿岸 | 地下室防淹（挡水板/潜水泵）、设备上移、应急撤离 |
| **沙尘暴** | 中东、北非、中国西北 | 空气过滤加强、室外设备防护、清洗频率增加 |
| **暴雪/冰灾** | 寒带/温带 | 屋顶承重监测、融雪、管道防冻、出入口除冰 |
| **野火/山火** | 加州、澳洲、地中海 | 空气质量（PM2.5 爆表）→全回风模式、室外进风口关闭 |
| **火山灰** | 冰岛、日本、印尼 | 空气滤网快速堵塞、室外设备损坏 |
| **雷电** | 热带/亚热带 | 防雷接地、浪涌保护、弱电设备保护 |
| **极端高温** | 全球变暖趋势 | 冷量不足（设计工况被突破）、电网限电、设备过热停机 |
| **极端低温（寒潮）** | 寒温带 | 管道冻裂、供暖负荷飙升、设备无法启动 |
| **海啸** | 沿海地震带 | 低层疏散、防水封堵、电气隔离 |

**产品启示**：需要「灾害预案模式」引擎，按楼宇所在地自动配置可能的灾害类型和预案。

#### 海拔与特殊环境

| 环境 | 影响 |
|------|------|
| **沿海/高盐雾** | 金属设备腐蚀加速（不锈钢/镀锌都不够）、空调室外机寿命缩短 |
| **高海拔（>3000m）** | 制冷设备降容（空气稀薄）、水沸点降低、人体缺氧（需补氧）|
| **地下建筑** | 无自然采光、通风全靠机械、排水靠泵、消防困难 |
| **水上建筑/邮轮** | 晃动/振动、盐雾腐蚀、淡水制造、自给能源 |
| **极地/南极站** | 极寒（-50°C）、极昼极夜、完全自给、运输困难 |

## 维度二：建筑年代与改造状态（Building Age & Retrofit）

### 建筑代际分类

| 年代 | 典型特征 | 运营挑战 |
|------|---------|---------|
| **1950s 以前** | 无空调，自然通风设计，砖石/木结构 | 加装设备空间受限、结构承重限制、保护性建筑不能改 |
| **1960-1980s** | 气动控制，窗式/分体空调，简单配电 | 气动控制器无法联网、管道老化、石棉/铅漆风险 |
| **1980-2000s** | 早期 DDC，Lonworks/私有协议，中央空调普及 | 厂商停产/无支持、协议过时、控制器算力低 |
| **2000-2015** | BACnet/Modbus 普及，有 BMS 但功能有限 | 系统可联网但数据质量差、传感器精度低 |
| **2015-至今** | IoT/云/AI，智能化程度高 | 系统多但集成差、数据孤岛、网络安全风险 |
| **新建（Greenfield）** | 从零设计，可按最佳实践配置 | 调试期长、系统磨合 |

### 改造场景分类

| 改造类型 | 说明 | 产品需求 |
|---------|------|---------|
| **绿地新建（Greenfield）** | 从零配置，最理想 | 标准化部署模板 |
| **棕地改造（Brownfield）** | 已有系统，加装/替换 | 老旧协议适配、增量部署 |
| **覆盖式（Overlay）** | 不动原系统，加一层 IoT 传感器 | 独立传感器管理、与旧系统数据融合 |
| **部分替换** | 替换核心控制器，保留末端 | 新旧系统并行运行期 |
| **历史保护建筑** | 不能改外观、不能打孔、不能走明线 | 无线传感器、隐蔽安装方案 |

**产品启示**：
- 绝大多数客户是 **Brownfield**（改造），不是新建
- 必须支持「新旧系统并存」— 一部分点位来自 BACnet BMS，一部分来自新加的 IoT 传感器
- 历史建筑是高价值客户（博物馆/古建筑/欧洲教堂），但安装限制极大

## 维度三：所有权与商业模式（Ownership & Business Model）

### 所有权类型

| 类型 | 决策者 | 投资逻辑 | 对产品的影响 |
|------|--------|---------|-------------|
| **业主自用** | 企业/机构自身 | 长期持有，关注运营成本和员工体验 | 功能全面，愿意投资 |
| **开发商持有出租** | 资产管理公司 | 关注租金收益和资产估值 | 节能=增加净收入，ESG 评分影响融资 |
| **REIT（房地产信托）** | 基金管理人 | 季度财报压力，关注 NOI（净营业收入）| 需要财务维度的能耗报告 |
| **政府/公共机构** | 政府采购 | 预算制，节能有政策考核 | 合规报表优先，采购流程长 |
| **PPP（公私合营）** | 社会资本方 | 运营期利润最大化 | 全生命周期成本优化 |
| **连锁品牌（加盟）** | 总部+加盟商 | 总部管标准，加盟商管执行 | 总部集中监控+门店本地管理 |
| **物业管理公司代管** | 物业公司 | 按合同管理，不关心资产增值 | SLA 考核、工单响应时间、投诉率 |
| **共有产权（业委会）** | 业主大会 | 预算紧张，决策慢 | 低成本方案，ROI 必须清晰 |

### 商业模式维度

| 运营模式 | 特点 | 产品需求 |
|---------|------|---------|
| **自持自管** | 一个主体，决策快 | 标准单楼宇管理 |
| **业主+物业** | 业主看报表，物业执行 | 双角色界面，审批流程 |
| **总部+门店（连锁）** | 标准化+本地化 | 多层级管理，模板下发，偏差告警 |
| **集团+区域+项目** | 三级管理 | 集团看对标，区域看总览，项目看细节 |
| **第三方托管** | FM 公司管理多个客户的楼 | 多租户隔离，SLA 仪表盘 |
| **节能服务（EMC/ESCO）** | 节能量分成 | 基线建立、节能量认证、M&V 报告 |

**产品启示**：
- 「连锁管理」是一个巨大市场（星巴克全球 35000 店、麦当劳 40000 店、万豪 8000 酒店）
- 需要支持「总部标准下发 → 门店执行 → 偏差告警 → 报表上卷」
- ESCO（节能服务）模式需要「基线+节能量计算」引擎

## 维度四：法规与合规（Regulations & Compliance）

### 按国家/地区的法规差异

| 法规领域 | 中国 | 美国 | 欧盟 | 中东 |
|---------|------|------|------|------|
| **建筑节能** | GB 50189 公建节能设计标准 | ASHRAE 90.1 | EPBD（建筑能效指令）| Estidama (阿布扎比) |
| **公共机构节能** | 《公共机构节能条例》| Executive Order 13834 | EU EED | - |
| **消防** | GB 50016 / GB 50116 | NFPA 各系列 | EN 各系列 | UAE Fire Code |
| **电气安全** | GB 50052 / GB 50054 | NEC (NFPA 70) | IEC 60364 | 各国引用 IEC |
| **无障碍** | GB 50763 | ADA | EN 17210 | UAE Accessibility Code |
| **室内空气** | GB/T 18883 | ASHRAE 62.1 | EN 16798 | Varies |
| **数据隐私** | 个人信息保护法(PIPL) | State laws (CCPA) | GDPR | PDPL (沙特) |
| **网络安全** | 等保 2.0 (等级保护) | NIST Cybersecurity | NIS2 Directive | NESA (阿联酋) |
| **绿建认证** | 绿色建筑三星 | LEED | BREEAM | Estidama Pearl |
| **碳排放** | 全国碳市场 | EPA / State programs | EU ETS | Varies |
| **医疗法规** | 《医疗机构管理条例》| Joint Commission / CMS | EU MDR | DHA/MOH |
| **食品安全** | GB 14881 | FDA / FSMA | EC 852/2004 | SFDA |

### 合规记录保存要求

| 数据类型 | 保存期限（典型） | 适用场景 |
|---------|----------------|---------|
| 能耗数据 | 3-5 年 | 所有公共建筑 |
| 温湿度记录 | 2-5 年 | 医院/药厂/食品/博物馆 |
| 冷链温度 | 3-5 年 | 医院/食品/物流 |
| 医气压力 | 3 年 | 医院 |
| 消防系统状态 | 永久 | 所有建筑 |
| 安防录像 | 30-90 天（一般）/ 3年+（关键区域）| 按场所和法规 |
| 门禁出入记录 | 1-3 年 | 写字楼/工厂/敏感场所 |
| 操作审计日志 | 3-5 年 | 所有控制操作 |
| 排放监测 | 3-5 年 | 工厂/锅炉房 |
| 审讯录像 | 永久 | 执法机构 |
| 电梯运行记录 | 4 年 | 所有有电梯的建筑 |
| 特种设备检测 | 设备寿命期 | 锅炉/电梯/压力容器 |

### 绿色建筑与 ESG

| 认证体系 | 运营期要求 | 产品需求 |
|---------|-----------|---------|
| **LEED O+M**（美国） | 能耗对标、IEQ 监测、垃圾分类率、绿色清洁 | 持续数据采集 + 年度报告 |
| **WELL**（全球） | 空气(7项)/水(6项)/光(7项)/热舒适(4项)/声环境(6项) | IEQ 全要素持续监测 |
| **绿色建筑三星**（中国） | 节能率、可再生能源比例、节水率 | 能耗数据 + 对标 |
| **BREEAM In-Use**（欧洲） | 资产性能 + 管理性能 | 双维度评估 |
| **NABERS**（澳大利亚） | 实际能耗评级（1-6星） | 能耗数据精确到分项 |
| **ESG 报告** | 碳排放(Scope 1/2/3)、水耗、废物 | 全自动数据归集+报告生成 |

**产品启示**：
- 合规不是可选项，是**许多客户的购买理由**
- 不同国家法规差异大 → 合规引擎需要可配置
- ESG 报告生成能力是面向投资者/上市公司的刚需
- WELL 认证带来的 IEQ 持续监测需求正在快速增长

## 维度五：能源来源与碳排放（Energy Source & Carbon）

### 能源供给类型

| 能源类型 | 说明 | 运营关注点 |
|---------|------|-----------|
| **电网供电** | 最普遍 | 电价（峰谷平）、需量管理、功率因数 |
| **天然气** | 锅炉/厨房/发电 | 价格波动、泄漏检测、排放监测 |
| **集中供热** | 市政热力 | 热量表计量、供回水温差、二级泵站管理 |
| **集中供冷** | 区域冷站 | 冷量计量、冷冻水参数 |
| **柴油** | 应急发电 | 油量监控、定期测试、排放 |
| **光伏（屋顶/立面）** | 自发自用/余电上网 | 发电量监控、逆变器状态、清洁维护 |
| **风力** | 分布式风电 | 发电量/风速、并网保护 |
| **地源/水源热泵** | 利用地温/水温 | 地下换热井水温、COP 衰减 |
| **储能（电池）** | 削峰填谷/光储互补 | 充放电管理、电池健康、安全 |
| **氢能** | 前沿探索 | 制氢/储氢/燃料电池 |
| **生物质** | 农村/特殊场景 | 燃料供应、排放 |

### 碳排放管理

```
Scope 1（直接排放）
├── 天然气锅炉燃烧
├── 柴油发电机运行
├── 制冷剂泄漏（R410A/R134a 等温室气体）
├── 公务车辆（如有）
└── 厨房燃气

Scope 2（间接排放 — 购入电力/热力）
├── 电网购电 × 电网排放因子
├── 购入蒸汽/热水
└── 购入冷冻水

Scope 3（价值链排放 — 更广范围）
├── 租户用电
├── 员工通勤
├── 建材生产
├── 废物处理
├── 水处理
└── 供应链
```

**产品启示**：
- 碳核算引擎需要：能耗数据 × 排放因子 → 自动计算
- 制冷剂泄漏追踪（Scope 1 重要组成部分）
- Scope 3 是趋势，越来越多企业要求
- 碳配额交易对接（中国碳市场/EU ETS）

## 维度六：人员与组织（People & Organization）

### 运营团队结构

| 角色 | 职责 | 工具需求 |
|------|------|---------|
| **设施总监/FM Director** | 战略规划、预算、供应商管理 | 管理驾驶舱、KPI 报表、对标分析 |
| **运维主管** | 日常管理、工单分配、巡检安排 | 工单系统、排班、巡检路线 |
| **暖通工程师** | HVAC 系统调试、故障排除 | 实时数据、趋势分析、参数调整 |
| **电气工程师** | 配电/发电/UPS 管理 | 电力监控、停电应急 |
| **弱电工程师** | 安防/消防/网络/门禁 | 视频、门禁、告警 |
| **一线维修工** | 现场维修、日常巡检 | 移动端工单、扫码报修 |
| **保安** | 安全巡逻、监控值班 | 视频墙、巡更系统、告警接收 |
| **保洁** | 清洁服务 | 任务排程、耗材管理 |
| **前台/客服** | 租户/客人服务接口 | 报修受理、投诉跟踪 |
| **第三方专业维保** | 电梯/消防/空调等专业维保 | 维保工单、进度跟踪、验收 |

### 人员配置模式

| 模式 | 适用场景 | 特点 |
|------|---------|------|
| **驻场团队** | 大型单体建筑 | 响应快，成本高 |
| **区域共享** | 连锁/多楼宇 | 一个团队管 3-5 栋楼，巡回服务 |
| **远程运维中心（ROC）** | 大规模管理 | 中心远程监控 + 本地最少人员 |
| **完全外包** | 中小型建筑 | FM 公司全包，按 SLA 考核 |
| **混合模式** | 大型综合体 | 核心团队驻场 + 专业分包 |

### 班次与值守

| 楼宇类型 | 值守模式 | 说明 |
|---------|---------|------|
| 写字楼 | 白班为主，夜间值守 | 工作日 8:00-18:00 全员，夜间/周末 1-2 人 |
| 酒店 | 三班倒 24h | 前台+工程+安保 全天候 |
| 医院 | 三班倒 24h | 设备科/安保 必须 24h |
| 数据中心 | 三班倒 24h | 或远程 + 紧急到场 |
| 商场 | 营业时段为主 | 10:00-22:00 全员，夜间安保 |
| 工厂 | 跟生产排班 | 生产停 = 维护窗口 |

**产品启示**：
- 「远程运维中心（ROC）」模式是你产品最直接对标的场景
- 需要支持排班管理（谁在值班 → 告警推给谁）
- 移动端（手机APP）对一线维修工是刚需
- 第三方维保商的协作界面（有限权限、工单对接）

## 维度七：供应链与采购（Supply Chain & Procurement）

### 设备与备件

```
设备全生命周期
├── 选型/采购（品牌/型号/参数/价格）
├── 安装/调试
├── 运行/监控
```

## 维度八：网络安全与 OT 安全（Cybersecurity & OT Security）

### 楼宇系统面临的网络威胁

| 威胁 | 风险 | 实际案例 |
|------|------|---------|
| **BMS 被入侵** | 远程篡改温度/关闭系统 | 2013 Target 数据泄露（通过 HVAC 供应商进入） |
| **勒索软件** | 锁定控制系统，勒索赎金 | 多起医院/工厂 OT 勒索事件 |
| **摄像头被劫持** | 隐私泄露、监控失效 | 大量 IP 摄像头暴露在公网 |
| **门禁系统被绕过** | 非授权进入 | 默认密码/未加密通信 |
| **DDC 控制器被利用** | 作为跳板攻击 IT 网络 | BACnet 设备无认证，可被扫描 |
| **数据泄露** | 人员出入/位置/视频 | GDPR/PIPL 违规罚款 |

### OT 安全分层

```
Level 5: 企业网络（IT）
    │
    │  [DMZ/防火墙]
    │
Level 4: 楼宇管理平台（你的产品）
    │
    │  [工业防火墙/网闸]
    │
Level 3: 楼宇控制网络（BACnet/Modbus）
    │
Level 2: 现场控制器（DDC/PLC）
    │
Level 1: 传感器与执行器
    │
Level 0: 物理过程（温度/压力/流量）
```

### 安全设计要求

| 层面 | 要求 |
|------|------|
| **网络分段** | IT 网络和 OT 网络必须隔离 |
| **认证授权** | 每个操作都有身份认证和权限校验 |
| **通信加密** | 边缘网关到云平台 TLS 加密 |
| **审计日志** | 所有控制操作留痕，不可篡改 |
| **固件更新** | 安全的 OTA 机制，防止恶意固件 |
| **漏洞管理** | 定期扫描、补丁管理 |
| **应急响应** | 网络攻击时的降级运行方案 |
| **数据脱敏** | 人脸/门禁/位置数据的隐私保护 |

**产品启示**：
- OT 安全是客户采购时的关键考量（尤其医院/政府/金融/军事）
- 你的产品本身就是攻击面（如果被攻破，能控制所有楼宇）
- 等保 2.0 / NIS2 / IEC 62443 合规可能是准入门槛

### 运营成本结构

```
楼宇运营总成本 (OPEX)
├── 能源费用 (30-50%)
│   ├── 电费（峰/谷/平/尖）
│   ├── 燃气费
│   ├── 供热/供冷费
│   └── 水费
├── 人工费用 (20-35%)
│   ├── 自有团队
│   └── 外包服务
├── 维修维保 (10-20%)
│   ├── 日常维修（材料+人工）
│   ├── 定期保养合同
│   └── 大修/更新
├── 耗材 (5-10%)
│   ├── 滤网/灯管/清洁用品
│   └── 办公用品
├── 保险 (2-5%)
├── 物业管理费/税费
└── 其他（软件许可/通信/水处理药剂等）
```

### 关键财务指标

| 指标 | 英文 | 说明 | 产品如何支持 |
|------|------|------|-------------|
| **EUI** | Energy Use Intensity | kWh/m²/年 | 能耗对标的基础 |
| **成本/m²** | Cost per sqm | 运营成本效率 | 对标排名 |
| **成本/人** | Cost per occupant | 人均运营成本 | 需要人数数据 |
| **SLA 达成率** | SLA Compliance | 工单响应/解决时间 | 工单系统自动统计 |
| **设备可用率** | Equipment Uptime | 设备正常运行时间占比 | 故障记录+运行时间 |
| **MTBF** | Mean Time Between Failures | 平均故障间隔 | 故障统计 |
| **MTTR** | Mean Time To Repair | 平均修复时间 | 工单时间统计 |
| **投诉率** | Complaint Rate | 租户/用户投诉频率 | 投诉管理模块 |
| **ROI** | Return on Investment | 改造投资回报 | 节能量计算 |
| **Payback Period** | 投资回收期 | 几年收回改造成本 | 投资分析工具 |
| **TCO** | Total Cost of Ownership | 全生命周期成本 | 设备 LCC 分析 |
| **NOI** | Net Operating Income | 净营业收入（REIT 关注） | 收入-运营成本 |

### 电价结构（影响策略的关键）

| 计价方式 | 说明 | 产品需求 |
|---------|------|---------|
| **分时电价** | 峰/谷/平/尖 不同价格 | 用电移峰策略、蓄冷/蓄热调度 |
| **需量电价** | 按最大需量（kW）收取基本电费 | 需量预警、错峰启动、负载均衡 |
| **阶梯电价** | 用量越多单价越高 | 用量预测、节能告警 |
| **实时电价（现货）** | 价格每 15 分钟变化 | 实时响应、储能充放电优化 |
| **容量电价** | 按申请容量收取 | 容量优化、减容建议 |

**产品启示**：
- 「省了多少钱」是最有说服力的 ROI 证明
- 电价策略引擎（按当地电价结构优化运行策略）是高价值功能
- 财务报表/成本分析是面向 FM Director 和 CFO 的功能

## 维度十一：数据治理与隐私（Data Governance & Privacy）

### 数据分类

| 数据类型 | 敏感度 | 示例 | 处理要求 |
|---------|--------|------|---------|
| **设备运行数据** | 低 | 温度/阀门/电流 | 正常存储，用于分析 |
| **能耗数据** | 低-中 | 各回路用电量 | 可能涉及租户隐私（用电时段暴露办公行为） |
| **门禁出入记录** | 中 | 谁在什么时间进出哪里 | 隐私数据，需要访问控制 |
| **视频监控** | 高 | 人脸/行为 | 严格合规（GDPR/PIPL） |
| **人员定位** | 高 | 实时位置追踪 | 必须有合法依据和告知义务 |
| **健康数据** | 极高 | 生命体征/就医记录 | 医疗级隐私保护 |
| **生物特征** | 极高 | 人脸/指纹/虹膜 | 单独同意、加密存储、到期删除 |

### 数据所有权问题

| 场景 | 谁拥有数据？ | 争议点 |
|------|------------|--------|
| 业主自管 | 业主 | 无争议 |
| 物业代管 | 业主（物业是数据处理方） | 物业更换时的数据交接 |
| 第三方 BMS | 业主（但厂商可能锁定数据） | 厂商是否开放数据导出？ |
| SaaS 平台（你） | 业主（你是处理方/托管方） | 数据导出/迁移/删除权利 |
| 租户数据 | 租户自身 | 业主能看到租户的详细用电行为吗？ |

### 数据生命周期

```
采集 → 传输 → 存储 → 处理 → 展示 → 归档 → 删除

每个环节的安全要求：
- 采集：传感器数据完整性（防篡改）
- 传输：TLS 加密
- 存储：加密存储、访问控制、备份
- 处理：最小权限原则
- 展示：数据脱敏（摄像头人脸模糊/门禁记录匿名化）
- 归档：压缩存储、冷存储
- 删除：到期自动删除、彻底删除（不可恢复）
```

**产品启示**：
- 数据治理能力是企业客户的采购门槛
- 需要支持「数据导出/迁移」— 客户不希望被锁定
- 视频/人脸/门禁数据要特别处理（GDPR 罚款可达全球营收 4%）
- 多租户数据隔离必须严格

## 维度十二：连锁与规模化管理（Portfolio & Multi-Site）

### 规模化带来的独特需求

| 需求 | 说明 |
|------|------|
| **标准化** | 总部定义标准 → 下发到所有门店/楼宇 |
| **偏差检测** | 某栋楼偏离标准 → 自动告警 |
| **对标排名** | 同类楼宇能耗/成本/满意度排名 |
| **集中采购** | 基于全部门店的设备/耗材需求统一采购 |
| **最佳实践复制** | A 店的优化策略 → 快速复制到 B/C/D 店 |
| **全局视图** | 一个屏幕看所有楼宇的关键指标 |
| **批量操作** | 一键修改所有门店的假日排程 |
| **异常定位** | 全局排名中的异常值 → 定位到具体楼宇/设备 |

### 连锁管理层级

```
集团总部
├── 区域公司 / 大区
│   ├── 城市公司 / 分区
│   │   ├── 项目/门店 A
│   │   ├── 项目/门店 B
│   │   └── ...
│   └── ...
└── 职能线
    ├── 工程部（看所有楼宇设备状态）
    ├── 能源部（看所有楼宇能耗）
    ├── 品质部（看所有楼宇 SLA/投诉）
    └── 财务部（看所有楼宇成本）
```

### 连锁典型客户规模

| 客户类型 | 典型楼宇数量 | 管理特点 |
|---------|------------|---------|
| 连锁酒店集团 | 500-10000 家 | 品牌标准 + 区域差异 |
| 连锁餐饮 | 1000-50000 家 | 极致标准化，每家店很小 |
| 连锁便利店 | 5000-80000 家 | 冷柜是核心设备 |
| 商业地产商 | 50-500 个项目 | 每个项目都是综合体 |
| 银行网点 | 1000-20000 个 | ATM 区 24h + 营业厅 |
| 连锁药店 | 500-10000 家 | 药品冷链是关键 |
| 加油站/充电站 | 1000-50000 个 | 防爆/安全为主 |
| 电信基站/机房 | 10000-500000 个 | 极致简化、远程管理 |

**产品启示**：
- 连锁市场是 SaaS 模式最适合的市场（标准化+规模化）
- 「一键策略下发」+ 「全局对标排名」是核心卖点
- 超大规模（5 万+ 站点）需要考虑架构性能
- 每个行业的「标准模板」不同，需要可配置

- 「Net Zero 路径规划」是越来越多客户的需求
- 碳核算 → 减碳措施建议 → 效果验证 → 闭环
- 水/废物/材料循环数据也要纳入

## 维度十四：施工与临时状态（Construction & Temporary States）

### 楼宇并非总在「正常运营」状态

| 状态 | 场景 | 运营挑战 |
|------|------|---------|
| **新建施工末期** | 系统调试、逐步交付 | 部分系统运行、部分在调试 |

- 楼宇状态不只是「正常/故障」，还有这些过渡态
- 「长期闲置楼宇」的管理（防冻、水质、定期巡检）是实际需求
- 物业交接时的系统数据迁移/交接流程

| **午休** | 1.5-2h（很多人需要午睡） | 0.5-1h | 2-3h（部分地区下午关门） | 1h | 1-2h（南欧更长） |
| **温度偏好** | 24-26°C（接受度高） | 21-23°C（偏冷） | 18-22°C（冷气开得很足） | 25-28°C（节能文化） | 22-24°C |
| **节假日** | 春节7天/国庆7天/调休 | 感恩节/圣诞/独立日 | 斋月（工作时间大变）/开斋节 | 黄金周/盂兰盆节 | 圣诞2-3周/复活节 |
| **宗教影响** | 低 | 中 | 极高（祷告时间/斋月） | 低 | 中低 |

### 特殊文化需求

| 需求 | 说明 | 影响系统 |
|------|------|---------|
| **斋月模式** | 穆斯林斋月期间工作时间缩短、晚间活动增加 | 空调/照明排程完全改变 |
| **祷告时间** | 每天 5 次，需要祈祷室 | 祈祷室空调/照明按祷告时间联动 |
| **午睡文化** | 中国/西班牙等 | 午休时段照明/温度调整 |
| **鞋文化** | 日本/中东脱鞋进入 | 换鞋区设计、地面清洁方式不同 |
| **分餐/合餐** | 影响食堂/餐厅设计 | 通风/空间规划不同 |
| **隐私观念** | 中东对女性空间隔离要求 | 独立的女性区域空调/门禁 |
| **风水/方位** | 中国/东南亚建筑布局偏好 | 可能影响设备安装位置 |
| **安息日** | 犹太教周五日落至周六日落不操作电器 | 需要全自动化排程 |

**产品启示**：
- 排程引擎必须支持「伊斯兰历/农历/公历」混合日历
- 斋月模式需要特殊处理（每年日期不同，每天祷告时间不同）
- 中国春节调休规则极其复杂（调哪天班/放哪天假，每年不同）
- 温度默认值应该按地区/文化可配置

## 维度十六：声学与振动（Acoustics & Vibration）

### 噪音源与控制

| 噪音源 | 场景 | 控制要求 |
|--------|------|---------|
| HVAC 设备（风机/压缩机/水泵） | 所有楼宇 | 机房隔振、风管消声、设备选型 NC 值 |
| 电梯 | 住宅/酒店/写字楼 | 曳引机减振、导靴润滑、轿厢隔音 |
| 外部交通（道路/铁路/飞机） | 临路/临轨/机场附近 | 隔声窗/墙、新风进口消声 |
| 施工噪音 | 改造期间 | 施工时间限制、隔声围挡 |
| 邻居/租户间 | 住宅/酒店 | 楼板隔声、墙体隔声 |
| 设备间对相邻房间 | 所有楼宇 | 浮筑楼板、弹性支撑 |

### 标准要求

| 场所 | 噪音等级(NC) | 说明 |
|------|------------|------|
| 音乐厅 | NC 15-20 | 极静 |
| 卧室/病房 | NC 25-30 | 安静 |
| 办公室 | NC 30-40 | 一般 |
| 餐厅/大堂 | NC 40-45 | 允许较高 |
| 商场/超市 | NC 45-55 | 嘈杂可接受 |

### 振动控制

| 场景 | 要求 |
|------|------|
| 半导体 Fab | VC-A 到 VC-G（纳米级） |
| 手术室 | VC-A |
| 精密实验室 | VC-C 到 VC-E |
| 住宅 | 可感知但不影响生活 |
| 办公 | 不影响打字/阅读 |

**产品启示**：
- 噪音监测作为 IEQ 的一部分，是 WELL 认证要求
- 对音乐厅/剧院/录音棚等场所，噪音是第一优先级
- 振动监测对数据中心（硬盘）、实验室（电子显微镜）是关键

## 维度十八：灾备与业务连续性（Disaster Recovery & Business Continuity）

### BCP 等级

| 等级 | 恢复时间目标(RTO) | 适用场景 | 设施要求 |
|------|------------------|---------|---------|
| **最高** | < 10 秒 | 手术室/ICU/数据中心/指挥中心 | 双路电源+UPS+发电机，无缝切换 |
| **高** | < 1 小时 | 酒店/商场/写字楼核心区域 | 发电机+部分 UPS |
| **中** | < 4 小时 | 学校/普通办公 | 发电机覆盖关键系统 |
| **低** | < 24 小时 | 仓库/停车场 | 基本应急照明 |

### 冗余设计

```
电力冗余
├── N：无冗余（单路电源）
├── N+1：一台备用（最常见）
├── 2N：完全双路独立（数据中心/医院）
└── 2(N+1)：最高冗余（Tier IV 数据中心）

冷源冗余
├── N+1 冷机（最常见）
├── 不同冷源互补（冷机+自然冷却+储冷）
└── 2N（极端场景）

网络冗余
├── 双 ISP 接入
├── 双核心交换
└── 带外管理通道（4G/卫星）
```

### 应急预案体系

```
├── 火灾应急预案
├── 停电应急预案
├── 停水应急预案
├── 电梯困人应急预案
├── 自然灾害应急预案（台风/地震/暴雨）
├── 安全威胁应急预案（入侵/暴力/恐怖）
├── 公共卫生应急预案（传染病/食物中毒）
├── 危化品泄漏应急预案（工厂/实验室）
├── 大面积停暖应急预案（寒冷地区）
├── 网络安全应急预案
└── 大规模人群疏散预案（场馆/交通枢纽）
```

**产品启示**：
- 应急预案管理（预案编辑、演练记录、触发联动）是高价值功能
- 不同楼宇类型需要不同的预案模板
- 预案与自动化联动（火灾报警 → 自动执行消防预案中的联动动作）

## 维度总结：交叉矩阵

每一栋楼的运营，实际上是这些维度的组合：

```
一栋楼的运营 = f(
    楼宇类型 × 子类型,        # 酒店 → 五星级度假
    子系统组合,                # HVAC + 照明 + GRMS + 泳池 + ...
    气候区,                   # 热带海岛
    建筑年代,                 # 2015年建成
    所有权,                   # REIT 持有 + 国际品牌管理
    法规环境,                 # 当地消防法规 + 品牌标准 + LEED 认证
    能源结构,                 # 电网 + 柴油发电机 + 屋顶光伏
    组织模式,                 # 驻场 20 人 + 远程监控中心
    规模（单体 vs 连锁）,     # 全球 3000 家酒店之一
    文化/地区习惯,            # 斋月期间作息完全变化
    安全等级,                 # 旅游区，反恐等级中等
    数据治理要求,             # GDPR（欧洲客人数据）
    可持续发展目标,           # 2030 年碳中和承诺
    当前状态                  # 正常运营 / 某层在翻新
)
```

这就是为什么楼宇运营这么复杂 — 不是一个维度的问题，是十几个维度的排列组合。

你的产品要做的，就是提供一个足够灵活的框架，让每一栋楼都能在这个框架里找到自己的位置。
